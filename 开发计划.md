# 四时有序 · 六爻解卦 —— 全栈演进规划

## Context

当前项目是一个纯前端的六爻解卦工具（GitHub Pages 托管），需要逐步演进为一个具备用户系统、数据持久化、邮件通知等能力的完整 Web 应用。

**核心决策**：
- 后端技术栈：Java Spring Boot
- AI 模型：DeepSeek（后端内置，用户无需自备 API Key）
- 计费策略：先免费上线，后续根据用户量决定
- 部署：云服务免费层

## 当前状态

- **前端**：纯 HTML/CSS/JS，无框架，PWA，~15 个源文件
- **数据存储**：localStorage（归档、AI 配置）
- **AI**：浏览器直连 API（需用户自备 Key，CORS 限制明显）
- **部署**：GitHub Pages（静态托管）
- **代码仓库**：`github.com/junxuezha1/do-not-behind-this-world`

## 分阶段实施计划

### 第一阶段：后端基础 + 内置 AI 服务（2-3周）

**目标**：建立 Spring Boot 骨架，内置 DeepSeek AI 服务，用户打开网页即可使用 AI 断卦，无需任何配置。

#### 1.1 Spring Boot 项目初始化
- 创建 Spring Boot 3.x 项目（Maven），放在 `backend/` 目录
- 模块划分：`controller` / `service` / `model` / `config`
- 配置文件管理（`application.yml`，区分 dev/prod）
- 集成 Swagger/OpenAPI 文档

#### 1.2 内置 AI 服务（核心）
- **后端持有 DeepSeek API Key**，存储在服务端配置中，绝不暴露给前端
- `POST /api/ai/analyze` —— 接收卦盘数据，后端调用 DeepSeek API
- SSE 流式响应（`SseEmitter`），实时把 AI 输出推送给前端
- 前端只需发一个请求，完全不感知 AI Provider 和 Key
- 预留多模型切换能力（管理员可在后台切换 DeepSeek/GPT/Claude）

#### 1.3 防滥用机制
- **IP 级限流**：未登录用户每 IP 每天 5 次 AI 分析（用 Redis 或内存计数）
- **登录用户限流**：每天 20 次（后续可调整）
- **请求频率限制**：同一用户/IP 60秒内最多 1 次请求
- **后端配置项**（`application.yml`）：
  ```yaml
  ai:
    provider: deepseek
    api-key: ${DEEPSEEK_API_KEY}  # 环境变量注入
    model: deepseek-chat
    max-tokens: 4096
    rate-limit:
      anonymous-daily: 5
      user-daily: 20
      min-interval-seconds: 60
  ```

#### 1.4 前端改造
- **移除**整个 AI 设置模态框（用户不再需要配置 API Key）
- **移除** `AI_PROVIDERS` 多 Provider 切换逻辑
- **简化** AI 分析：一个按钮即可，无需选择 Provider
- `streamAIAnalyze()` 改为调用后端 `/api/ai/analyze`
- 显示剩余次数提示（如"今日剩余 3/5 次"）
- 超出限额时友好提示（"今日免费次数已用完，注册登录可获得更多次数"）

#### 1.5 本地开发环境
- 前端：`python -m http.server 8080` 或 Vite dev-server
- 后端：`mvn spring-boot:run`（端口 8081）
- 跨域配置（WebMvcConfigurer 允许 localhost:8080）

**关键文件变更**：
- 新建 `backend/` Spring Boot 项目
- 修改 `js/app-bundle.js`：简化 AI 调用，移除 Provider 选择
- 修改 `index.html`：移除 AI 设置模态框、Provider 选择器
- 修改 `css/main.css`：移除设置相关样式

### 第二阶段：数据库 + 智能归档（2-3周）

**目标**：用数据库替代 localStorage，AI 自动生成归档摘要，用户可编辑确认。

#### 2.1 数据库设计
```sql
-- 归档表
CREATE TABLE gua_archive (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id    VARCHAR(100),         -- 未登录用户用 session 标识
  user_id       BIGINT,               -- 登录用户关联（第三阶段启用）
  question      VARCHAR(500),         -- 占问
  date_time     DATETIME,             -- 起卦时间
  gan_zhi       VARCHAR(100),         -- 四柱干支
  ben_gua       VARCHAR(50),          -- 本卦
  bian_gua      VARCHAR(50),          -- 变卦
  shi_yao       VARCHAR(50),          -- 世爻
  ying_yao      VARCHAR(50),          -- 应爻
  dong_yao      VARCHAR(200),         -- 动爻
  kong_wang     VARCHAR(20),          -- 日空亡
  raw_text      TEXT,                  -- 原始卦盘文本
  ai_result     TEXT,                  -- AI完整分析结果
  conclusion    TEXT,                  -- 结论摘要（AI生成，用户可改）
  judgment      ENUM('JI','XIONG','ZHONG') DEFAULT 'ZHONG', -- 吉凶判断
  ying_qi_text  VARCHAR(500),         -- 应期文字描述
  ying_qi_start DATE,                 -- 应期开始日期
  ying_qi_end   DATE,                 -- 应期结束日期
  verified      BOOLEAN DEFAULT FALSE,-- 是否已验证
  verify_result ENUM('ACCURATE','INACCURATE','PARTIAL','UNKNOWN'),
  verify_note   TEXT,                 -- 验证笔记
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- AI用量记录表
CREATE TABLE ai_usage_log (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id     BIGINT,
  ip_address  VARCHAR(50),
  tokens_used INT,
  model       VARCHAR(50),
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 2.2 归档 CRUD API
- `POST /api/archive` —— 创建归档
- `GET /api/archive` —— 列表（分页、搜索、按日期/卦名筛选）
- `GET /api/archive/{id}` —— 详情
- `PUT /api/archive/{id}` —— 更新（编辑结论、标记验证）
- `DELETE /api/archive/{id}` —— 删除

#### 2.3 AI 智能归档（用户最关心的功能）
- AI 分析完成后，后端额外调用一次 DeepSeek（用更短的 prompt）：
  - 输入：AI 完整分析结果
  - 输出 JSON：`{ conclusion, judgment, yingQiText, yingQiStartDate, yingQiEndDate }`
- 前端收到后展示归档草稿弹窗：
  - 结论摘要（可编辑的文本框）
  - 吉凶标记（吉/凶/中 三选一）
  - 应期时间（可编辑的日期选择器）
  - 「保存归档」按钮
- 这个额外调用使用小模型或短 prompt，成本很低

#### 2.4 前端归档改造
- 归档数据从 localStorage → API
- AI 分析完成后自动弹出智能归档草稿
- 归档列表页重做：卡片式展示，支持搜索筛选
- 归档详情页：完整卦盘 + AI 分析 + 验证功能
- 未登录用户用 sessionStorage 临时存储，提示登录以永久保存

**关键文件**：
- `backend/.../model/GuaArchive.java`
- `backend/.../controller/ArchiveController.java`
- `backend/.../service/ArchiveService.java`
- `backend/.../service/AiSummaryService.java`（AI 归档摘要生成）
- 前端归档相关代码重构

### 第三阶段：用户系统（2-3周）

**目标**：实现注册登录，归档数据与用户绑定，AI 用量按用户管理。

#### 3.1 用户表
```sql
CREATE TABLE sys_user (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  username      VARCHAR(50) UNIQUE,
  email         VARCHAR(100) UNIQUE,
  password_hash VARCHAR(255),         -- BCrypt
  nickname      VARCHAR(50),
  avatar_url    VARCHAR(500),
  role          ENUM('USER','VIP','ADMIN') DEFAULT 'USER',
  ai_quota      INT DEFAULT 20,       -- 每日 AI 次数配额
  email_verified BOOLEAN DEFAULT FALSE,
  created_at    TIMESTAMP,
  last_login    TIMESTAMP
);
```

#### 3.2 认证方案
- Spring Security + JWT
- 接口：注册、登录、刷新 Token、获取用户信息
- 后续扩展：微信扫码登录（适合国内用户）

#### 3.3 权限与数据隔离
- 未登录：可使用解析 + 规则引擎，AI 按 IP 限流（5次/天）
- 已登录 USER：AI 20次/天，归档云端同步
- VIP：更高 AI 配额，优先使用更强模型
- ADMIN：管理后台，查看用量统计

#### 3.4 前端改造
- 登录/注册页（支持邮箱注册）
- 顶部导航显示用户信息 + 剩余 AI 次数
- 登录后自动同步 localStorage 中的临时归档到云端

### 第四阶段：邮件通知 + 应期提醒（1-2周）

**目标**：应期到来时邮件提醒用户验证卜问结果。

#### 4.1 邮件服务
- Spring Boot Starter Mail + QQ邮箱 SMTP（免费）
- 邮件模板：
  - 注册验证邮件
  - 应期提醒邮件（含卦象摘要、AI 核心判断、一键验证链接）
  - 密码重置邮件

#### 4.2 定时扫描任务
- `@Scheduled(cron = "0 0 8 * * *")` 每天早8点
- 扫描所有 `ying_qi_start <= 今天 + 3天 AND verified = false` 的记录
- 给对应用户发邮件提醒
- 邮件包含验证链接，点击跳转到归档详情页

#### 4.3 应期日期解析增强
- 后端解析 AI 输出的干支应期为公历日期（如"午月" → 6月~7月）
- 前端日历组件标记应期日期范围

## 部署方案（云服务免费层）

| 组件 | 推荐方案 | 免费额度 |
|------|---------|---------|
| 前端 | Vercel / GitHub Pages | 无限静态站点 |
| 后端 | Railway / Render | Railway 500h/月; Render 750h/月 |
| 数据库 | Railway MySQL / Supabase PgSQL | 5GB 免费 |
| Redis | Railway Redis / Upstash | Upstash: 10K 命令/天免费 |
| 邮件 | QQ邮箱 SMTP | 免费 |
| AI | DeepSeek API | 按 token 计费，成本极低(~¥0.01/次) |

**预估月成本**（100 用户/天）：
- DeepSeek API：约 ¥30-100/月（取决于平均用量）
- 服务器/数据库：免费层足够
- 总计：约 ¥50-150/月

## 下一步行动（第一阶段启动）

1. 在项目根目录 `backend/` 创建 Spring Boot 项目
2. 实现 `/api/ai/analyze` 接口，内置 DeepSeek API Key，SSE 流式转发
3. 实现 IP 级限流（内存 Map 或 ConcurrentHashMap，暂不需要 Redis）
4. 前端简化：移除 AI 设置模态框和 Provider 选择，改为直接调用后端
5. 本地跑通：粘贴卦盘 → 解析 → 点击"AI断卦" → 后端代理调 DeepSeek → 流式返回
6. 后端部署到 Railway，前端更新 GitHub Pages

## 验证方式

- **第一阶段**：粘贴卦盘 → 解析 → AI断卦（自动通过后端） → 查看流式输出 → 确认限流生效
- **第二阶段**：AI分析 → 弹出智能归档草稿 → 编辑保存 → 刷新页面仍可见
- **第三阶段**：注册 → 登录 → 归档 → 退出 → 重新登录 → 历史归档仍在
- **第四阶段**：创建含应期归档 → 手动触发定时任务 → 收到邮件 → 点击验证
